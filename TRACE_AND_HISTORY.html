Trace Logs & Conversation History

This report illustrates how the Madadgar AI assistant orchestrates multiple agents to fulfill a user's home-service booking request. Each Turn below shows the user's message, the assistant's reply (with any rich UI elements like service cards or pills), and the corresponding agent trace logs captured on the backend — the same logs the in-app activity view surfaces.

Agents involved

AgentResponsibility
Orchestrator Agent Top-level coordinator. Decides which agents to invoke based on the extracted intent (smalltalk, missing fields, or complete service request).
Intent Agent Calls the LLM (OpenRouter) to extract { intentType, service, location, time, language } from the user's message + history. Deterministic keyword guardrail + heuristic fallback ensure resilience when the LLM fails.
Provider Discovery (frontend) Once intent is complete, the frontend queries Firestore for catalog services matching the inferred category, re-ranks by location overlap, and returns the top results as bookable cards.
Slot Validator (frontend) Validates that the requested time falls within the bookable window (10 AM–9 PM, next 30 days). If not, surfaces tappable pills with available slots and preserves the valid half across picks.
Booking Agent Persists the confirmed booking to Firestore and fires push notifications to the chosen provider via the Expo Push API.

Trace log schema

Every backend log entry is a JSON record of the form:

{
  timestamp: "2026-05-20T18:42:11.842Z",   // ISO 8601, set by backend
  agent:     "Orchestrator Agent",         // Authoring agent name
  message:   "Intent complete, returning suggestions context",
  data:      { ... }                        // Optional structured payload
}

The frontend's ChatContext attaches the full logs array returned by each /api/orchestration/chat response onto the corresponding assistant message, alongside the message's id, createdAt timestamp, and replyForUserId link to the user message that triggered it.
